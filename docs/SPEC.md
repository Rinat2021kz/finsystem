# ТЗ — Финансовая учётная система для малого бизнеса

Полное функциональное и техническое задание. Источники: инструкция по 4 Excel-файлам (кофейня, ОПУ Академии, инвестиции, фин-модель) + ранее согласованные решения проекта. Для разработки в Claude Code читать вместе с `CLAUDE.md` (правила и порядок работы) и `schema.sql` (БД).

---

## 1. Что за продукт

Управленческая финансовая система для малого бизнеса (не «онлайн-Excel»). Клиент вносит простые данные об операциях, а система **сама считает** управленческие отчёты: ДДС, ОПУ, баланс денег, отчёты по проектам, планы и инвестиционные модели.

Клиент не разбирается в формулах. Клиент понимает: сколько денег сейчас, откуда пришли и куда ушли, прибыльный ли бизнес, какие расходы съедают прибыль, какие проекты прибыльны, сколько нужно инвестиций и когда они окупятся, где кассовый разрыв.

Продукт двусторонний: это одновременно **сервис для собственника** и **рабочий кабинет финансового консультанта**, который ведёт нескольких клиентов, проверяет заполнение, комментирует, готовит отчёты и модели.

## 2. Целевая аудитория (значение по умолчанию)

Первичный сегмент — **проектный/услуговый малый бизнес и микробизнес Казахстана** (услуги, консалтинг, онлайн-образование, небольшие заведения, торговля), а также **финансовые консультанты**, которые их ведут. Валюта по умолчанию — тенге (KZT). Отраслевые шаблоны на старте: универсальный малый бизнес, услуги, онлайн-образование, кофейня/заведение, торговля, проектный бизнес.

## 4. Модули

### 4.1. Регистрация и компании
Пользователь ведёт одну или несколько компаний. При создании компании: название, отрасль (шаблон), валюта, дата начала учёта, часовой пояс, включён ли проектный учёт, включён ли инвестиционный модуль. Онбординг-мастер — см. раздел 8.

### 4.2. Справочники
Счета, категории доходов/расходов (с подкатегориями через `parent_id`), проекты, продукты/услуги, контрагенты, ответственные. Категории **настраиваемые** — нельзя жёстко зашивать только примеры из файлов. Стартовый набор создаётся из выбранного отраслевого шаблона.

### 4.3. Операции (главный экран)
Добавление дохода, расхода, перевода. Поля операции — см. `schema.sql` (transactions). Быстрые кнопки: «Добавить доход», «Добавить расход», «Перевести между счетами», «Добавить проект», «Добавить продажу», «Расход по проекту». Таблица операций с фильтрами по дате, счёту, категории, проекту.

### 4.4. ДДС
«Как двигались деньги». Начальный остаток, поступления, выплаты, переводы, конечный остаток; разрезы по счёту, категории, дням, месяцам. Формулы — раздел 6.1.

### 4.5. ОПУ
«Прибыльный бизнес или нет». Выручка, переменные, валовая прибыль, постоянные, операционная прибыль, налоги, проценты, амортизация (если включена), чистая прибыль, рентабельность. Полная структура, а не упрощённая. Формулы — раздел 6.2.

### 4.6. Баланс денег
Управленческий баланс денежных средств по счетам (не полный бухбаланс на старте): счёт, начальный остаток, поступления, списания, переводы, текущий остаток, доли счетов, динамика. Формулы — раздел 6.3.

### 4.7. Проекты
Для бизнеса заказами/проектами. Поля и статусы — см. `schema.sql`. Показатели: оплачено факт, долг, расходы проекта, плановая маржа, кассовая маржа, рентабельность. Формулы — раздел 6.4.

### 4.8. Планирование продаж
План по продуктам: цена, количество по месяцам, рост, сезонность, скидки, возвраты, канал. Формулы — раздел 6.5.

### 4.9. Планирование расходов
Форматы расхода: фиксированный, % от выручки, на единицу продаж, разовый, ФОТ, налоговая ставка, проценты по займам. Формулы — раздел 6.6.

### 4.10. Инвестиционный модуль
Потребность в инвестициях, план продаж, план расходов, финмодель, показатели инвестора (доля, дивиденды, ROI, срок окупаемости), графики. Формулы — раздел 6.7.

### 4.11. Дашборды (+ наш слой понимания)
Простой дашборд с ключевыми показателями (остаток денег, доходы/расходы за месяц, чистая прибыль, рентабельность, денежный поток, топ-5 доходов/расходов, остатки по счетам, динамика выручки/прибыли, проекты с долгами и низкой маржой, кассовый разрыв, срок окупаемости).

**Наш слой понимания на каждом виджете (из ранних решений):**
- светофор/алерты по показателю (красный–жёлтый–зелёный) с настраиваемыми порогами;
- комментарий консультанта прямо в виджете (расшифровка цифры словами);
- «что делать дальше» — рекомендация/следующий шаг;
- динамика к прошлому периоду + мини-тренд.

**Настройка (из ранних решений):** консультант задаёт стартовый дашборд под клиента; клиент прячет/показывает виджеты и меняет их размер/детализацию. Хранится в `dashboard_configs.config_json`. На дашборде **не должно быть Excel-ошибок** — вместо них «нет данных»/0/предупреждение.

### 4.12. Калькуляторы планирования (наше добавление; в файле их нет)
Отдельный лёгкий раздел «что если», который дополняет учёт и переиспользуется в обучении:
- **Юнит-экономика:** `маржа продажи = чек − переменные − CAC`; вердикт «масштабировать да/нет».
- **Точка безубыточности:** `постоянные / маржа`; для цели `(постоянные + цель) / маржа`.
- **CAC / LTV:** `CAC = бюджет / клиенты`; `LTV ≈ маржа × число покупок`; окупаемость клиента.
Единое определение маржи во всех калькуляторах. CAC — переключатель «на продажу / на клиента».

## 5. Модель данных

Полная схема — в `schema.sql`. Сущности: users, companies, company_members, accounts, categories, counterparties, products, projects, transactions, sales_plan, expense_plan, investment_models, investment_items, report_snapshots, audit_log, dashboard_configs, share_links.

Ключевое: деньги — целое в тиынах (BIGINT `*_minor`); `date_cashflow` (ДДС) ≠ `period_pnl` (ОПУ); переводы не доход/расход; расчётные величины не хранятся (кроме снимков закрытых месяцев).

## 6. Расчётный модуль (формулы)

Все формулы — на backend. Любое деление защищено (см. раздел 9).

### 6.1. ДДС за период
По всем счетам:
- `opening = Σ(начальные остатки счетов) + Σ(доходы до начала) − Σ(расходы до начала)`
- `cash_in = Σ операций income за период (по date_cashflow, include_in_cashflow)`
- `cash_out = Σ операций expense за период`
- переводы = 0 для сводного отчёта (не меняют общий остаток)
- `closing = opening + cash_in − cash_out`

По одному счёту: добавляются `transfer_in` и `transfer_out`; `closing = opening + cash_in − cash_out + transfer_in − transfer_out`.

### 6.2. ОПУ за месяц
- `revenue = Σ income, где period_pnl = месяц`
- `variable_expenses = Σ expense (переменные), period_pnl = месяц`
- `fixed_expenses = Σ expense (постоянные)`; `payroll`, `taxes`, `interest`, `depreciation` — аналогично по категориям
- `gross_profit = revenue − variable_expenses`
- `operating_profit = gross_profit − fixed_expenses − payroll`
- `net_profit = operating_profit − taxes − interest − depreciation`
- `profitability = net_profit / revenue` если `revenue > 0`, иначе `null`

### 6.3. Остаток по счёту
`current = opening + доходы_на_счёт − расходы_со_счёта + переводы_на_счёт − переводы_со_счёта`

### 6.4. Проект
- `paid_fact = Σ доходных операций проекта`
- `debt = contract_value − paid_fact`
- `project_expenses = Σ расходных операций проекта`
- `planned_margin = contract_value − project_expenses`
- `cash_margin = paid_fact − project_expenses`
- `planned_profitability = planned_margin / contract_value` если `contract_value > 0`
- `cash_profitability = cash_margin / paid_fact` если `paid_fact > 0`

### 6.5. План продаж
- `quantity[1] = ввод`; `quantity[n] = quantity[n−1] × (1 + growth_rate)`
- `revenue = price × quantity × seasonality_factor`
- `total_revenue_month = Σ по продуктам`
- количество продаж: по умолчанию округлять до целого для операционного плана; в прогнозе допускать дробь (настройка).

### 6.6. План расходов
- фиксированный = сумма; процентный = `выручка × percent`; на единицу = `количество × per_unit`; разовый = сумма в месяце; ФОТ/налог/проценты — по типу.

### 6.7. Инвестиции
- `total_investment = capex + launch + operating_buffer + reserve + other`
- `investor_share = investment_amount / company_value` если `company_value > 0`
- `investor_dividend = net_profit × dividend_policy_percent × investor_share` если `net_profit > 0`, иначе 0
- `cumulative_return = Σ investor_dividend по периодам`
- `ROI = cumulative_return / investment_amount − 1` если `investment_amount > 0`
- `payback_period = первый период, где cumulative_return ≥ investment_amount`; иначе «не окупается в горизонте»

## 7. Ключевые бизнес-правила (не нарушать)

1. **Разделять ДДС и ОПУ.** ДДС — по дате денег; ОПУ — по экономическому периоду. Смешение = неверные выводы.
2. **Переводы между счетами не доход и не расход.** Меняют только остатки счетов.
3. **Капитальные расходы — отдельно.** MVP: `capex_mode='simple'` (в ОПУ сразу, явно подписано). Точнее: в ДДС расход, в ОПУ — амортизация.
4. **Налоги — отдельный блок.** MVP: пользователь вводит налоги как статью расхода; авторасчёт по ставкам — позже.
5. **Одна валюта на компанию** (MVP). Мультивалютность и валютные счета — позже.
6. **Закрытие месяца.** После закрытия обычный пользователь не меняет операции; админ может открыть; хранится снимок; изменения логируются.
7. **Никаких Excel-ошибок** (#DIV/0!, #N/A, #VALUE!, #REF!, #NAME?). Все формулы с проверками.
8. **Округление.** Деньги — до тенге по умолчанию (хранение в тиынах). Количество — до целого (в прогнозе можно дробь). Проценты — 1–2 знака.
9. **История изменений.** Кто/когда/что изменил, было/стало (audit_log).
10. **Отчёты по выбранному периоду**, а не по текущей дате (никаких TODAY() в логике).

## 8. Сценарии клиента

**Первый вход (мастер):** название бизнеса → валюта → месяц старта учёта → счета → остатки на дату старта → основные доходы → основные расходы → нужен ли проектный учёт → нужна ли инвестмодель → нужно ли планирование продаж. Далее система создаёт стартовые справочники из шаблона.

**Ежедневно:** добавление доходов/расходов/переводов (быстрые кнопки).
**Еженедельно:** остатки, расходы/поступления недели, крупные платежи, просроченные долги по проектам, кассовые риски.
**Ежемесячно:** проверка операций, сверка остатков, ДДС, ОПУ, прибыль, анализ расходов, закрытие месяца.
**Проекты:** создать проект → привязать оплаты/расходы → видеть долг/оплату/затраты/маржу/рентабельность.
**Инвестиции:** создать сценарий → потребность, план продаж/расходов, оценка, доля инвестора, ROI, срок окупаемости, графики.

## 9. Ошибки и валидации

- деление на ноль → `null` / 0 / «нет данных» / понятное предупреждение (никогда не ошибка);
- отрицательные суммы в операциях запрещены (знак задаёт тип);
- перевод требует обоих счетов и не пишет в доход/расход;
- нельзя менять операции закрытого месяца (кроме админа с логом);
- при пустых данных графики показывают «нет данных».

## 10. Экспорт

MVP: экспорт отчётов (ДДС, ОПУ, баланс денег) и данных в **Excel** и **PDF**, по выбранному периоду. Инвест-модель и графики — экспорт в PDF (задел под питч-дек в v3).

## 11. Админка, закрытие месяца, аудит

Закрытие/открытие месяца, снимки (`report_snapshots`), журнал действий (`audit_log`), управление участниками и ролями, генерация клиентских ссылок (`share_links`).

## 12. Безопасность (базовая)

Мультитенантность: любая выборка — по `company_id`, проверка членства в `company_members`. Хранение паролей хешем. HTTPS. Логирование действий. Экспорт и удаление данных аккаунта. Резервные копии БД. Клиентские ссылки — случайный токен, срок действия, отзыв.

## 13. Этапы

### MVP (первый рабочий продукт)
Регистрация → компания → счета → категории → операции → переводы → ДДС по месяцам → ОПУ по месяцам → остатки по счетам → простой дашборд (+ базовый слой понимания) → экспорт Excel/PDF → защита от ошибок → выбор периода. Одна валюта. Налоги — статьёй расхода. Капзатраты — упрощённо.

Отложить: импорт выписок, налоговый калькулятор, полный бухбаланс, сложная инвест-аналитика, мультивалютность, интеграции банков, распознавание чеков, сложные права.

### v2
Проекты и отчёт по проекту, долги и маржинальность проектов, план продаж, план расходов, прогноз прибыли, сравнение план/факт, комментарии консультанта, приглашение консультанта в аккаунт клиента, конструктор клиентского дашборда + клиентские ссылки, калькуляторы планирования (4.12).

### v3
Инвестмодели, оценка компании, сценарии (пессим/база/оптим), ROI/дивиденды/доли, графики для питч-дека, экспорт инвест-модели, сравнение сценариев. Плюс: импорт Excel/CSV/выписок, Telegram/WhatsApp-бот, распознавание чеков, интеграции Kaspi/Halyk (если возможно), уведомления о кассовом разрыве, рекомендации по данным.

## 14. Приёмочные тесты (должны проходить)

1. **Доход** 100 000 на Kaspi → ДДС +100 000, остаток Kaspi +100 000, ОПУ выручка +100 000 (если указан месяц), баланс вырос.
2. **Расход** 30 000 с Kaspi → ДДС −30 000, Kaspi −30 000, ОПУ расход +30 000 (если учитывается), прибыль ниже.
3. **Перевод** 50 000 Kaspi→Halyk → Kaspi −50 000, Halyk +50 000, общий остаток без изменений, ОПУ без изменений.
4. **Нулевая выручка**, есть расходы → прибыль отрицательная, рентабельность без ошибки («нет выручки для расчёта»).
5. **Проект с долгом:** стоимость 500 000, оплачено 200 000, расходы 100 000 → долг 300 000, плановая маржа 400 000, кассовая маржа 100 000.
6. **Инвестиции:** 10 000 000 при оценке 40 000 000 → доля инвестора 25%.
7. **ROI при инвестициях = 0** → ROI не считается, деления на ноль нет.
8. **Закрытый месяц:** обычный пользователь не может менять операцию; админ может открыть; действие в журнале.
9. **Капзатраты** 1 000 000 → ДДС показывает расход; ОПУ по настройке (сразу или амортизация).
10. **Выбор периода** март 2026 → все отчёты за март 2026, независимо от даты компьютера.

## 16. Технологический стек (рекомендация, можно заменить)

- **Frontend + Backend:** Next.js (App Router) + TypeScript — один кодовый репозиторий, API-роуты + интерфейс.
- **БД:** PostgreSQL. **ORM:** Prisma (миграции из `schema.sql`).
- **Аутентификация:** Auth.js (email/пароль или magic-link); роли — в `company_members`; клиент — по `share_links` (токен в URL, без аккаунта).
- **Расчётный модуль:** серверные TS-модули (`/lib/calc/*`) с юнит-тестами (Vitest/Jest) на приёмочные сценарии раздела 14.
- **Отчёты/экспорт:** генерация Excel (напр. exceljs) и PDF (напр. @react-pdf/renderer или Playwright-print).
